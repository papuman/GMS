**Critical Migration Requirement: Consecutive Numbering** ‚ö†Ô∏è

**The Law:**
When a business switches e-invoicing providers, they **MUST continue** their consecutive invoice numbering from the last number used. They **CANNOT** restart at 1.

**20-Digit Consecutive Structure:**
```
AAA-BBBBB-CC-DDDDDDDDDD
‚îÇ   ‚îÇ     ‚îÇ  ‚îî‚îÄ Consecutive number (10 digits): 0000000001 to 9999999999
‚îÇ   ‚îÇ     ‚îî‚îÄ Document type (2 digits): 01=Invoice, 02=Debit, 03=Credit, 04=Ticket
‚îÇ   ‚îî‚îÄ Terminal/point-of-sale (5 digits): 00001 to 99999
‚îî‚îÄ Establishment/branch (3 digits): 001 to 999

Example: 001-00001-01-0000000152
         Branch 1, Terminal 1, Invoice type, Invoice #152
```

**Migration Scenarios:**

| Scenario | Consecutive Handling | Special Requirements |
|----------|----------------------|----------------------|
| **First-time (from paper)** | Start at 00000000001 | Register in ATV/OVI portal first |
| **Switching providers** | Continue from last number | Must know exact last number used |
| **Adding new branch/terminal** | New branch starts at 00000000001 | Branch code differs (002, 003, etc.) |
| **Reached max (9,999,999,999)** | Can restart at 00000000001 | Extremely rare case |
| **Version 4.3 to 4.4** | Continue same sequence | Add provider field to XML |

**Critical Migration Risks:**

**HIGH RISK:**
- ‚ö†Ô∏è **Duplicate consecutive numbers** ‚Üí Invoice REJECTED by Hacienda
- ‚ö†Ô∏è **Invalid digital certificate** ‚Üí Cannot submit invoices
- ‚ö†Ô∏è **Version mismatch** ‚Üí Non-compliance with v4.4

**MEDIUM RISK:**
- ‚ö†Ô∏è **Customer confusion** ‚Üí Communication plan needed
- ‚ö†Ô∏è **Rejected invoices** ‚Üí User error during setup
- ‚ö†Ô∏è **Data quality issues** ‚Üí Clean import data first

**LOW RISK:**
- ‚ö†Ô∏è **PDF formatting** ‚Üí Cosmetic issues only
- ‚ö†Ô∏è **Timezone issues** ‚Üí Usually auto-handled

**Implementation Code Pattern:**
```python
# Consecutive number management for Costa Rica
consecutive_sequences = {
    '01': {'last_used': 152, 'next': 153},      # Invoices (Facturas)
    '02': {'last_used': 8, 'next': 9},          # Debit Notes
    '03': {'last_used': 45, 'next': 46},        # Credit Notes
    '04': {'last_used': 2340, 'next': 2341},    # Tickets (Tiquetes)
    '05': {'last_used': 0, 'next': 1},          # Export Invoices
    # ... other document types
}

def get_next_consecutive(establishment, terminal, doc_type):
    """
    Generate next consecutive number for invoice
    Must be atomic to prevent duplicates
    """
    sequence_key = f"{establishment}-{terminal}-{doc_type}"

    with transaction.atomic():
        sequence = ConsecutiveSequence.objects.select_for_update().get(
            establishment=establishment,
            terminal=terminal,
            doc_type=doc_type
        )
        next_number = sequence.last_used + 1
        sequence.last_used = next_number
        sequence.save()

    return f"{establishment}-{terminal}-{doc_type}-{next_number:010d}"
```

**Key Compliance Requirements:**

**1. Version 4.4 Mandatory (Since September 1, 2025)**
- v4.3 is DEPRECATED
- All invoices must use v4.4 XML schema
- Provider identification field now REQUIRED

**2. Consecutive Numbering (No Exceptions)**
- 20-digit format strictly enforced
- No gaps allowed in sequence
- No duplicates (instant rejection)
- Must continue when switching providers

**3. Digital Signature (X.509 Certificate)**
- Valid certificate from Hacienda-approved CA
- Certificate must not be expired
- Private key must be secured
- Signature must validate against public key

**4. Mandatory XML Fields (v4.4):**
```xml
<Invoice>
  <Clave>50-digit unique key (generated by Hacienda algorithm)</Clave>
  <Fecha>ISO 8601 timestamp</Fecha>
  <Emisor>
    <Identificacion>
      <Tipo>01=F√≠sica, 02=Jur√≠dica, 03=DIMEX, 04=NITE</Tipo>
      <Numero>Tax ID number</Numero>
    </Identificacion>
    <Nombre>Business legal name</Nombre>
    <NombreComercial>Trade name (optional)</NombreComercial>
    <Ubicacion>
      <Provincia>1-7</Provincia>
      <Canton>01-99</Canton>
      <Distrito>01-99</Distrito>
    </Ubicacion>
  </Emisor>
  <Receptor>
    <!-- Customer identification -->
  </Receptor>
  <LineaDetalle>
    <Codigo>CABYS 13-digit code</Codigo>
    <Descripcion>Product/service description</Descripcion>
    <PrecioUnitario>Unit price</PrecioUnitario>
    <MontoTotal>Line total</MontoTotal>
    <Impuesto>
      <Codigo>01=IVA</Codigo>
      <Tarifa>4 (for 4% IVA)</Tarifa>
      <Monto>Tax amount</Monto>
    </Impuesto>
  </LineaDetalle>
  <ResumenFactura>
    <TotalVenta>Sale total</TotalVenta>
    <TotalImpuesto>Total tax</TotalImpuesto>
    <TotalComprobante>Invoice total</TotalComprobante>
  </ResumenFactura>
  <InformacionReferencia>
    <!-- NEW in v4.4: Provider identification -->
    <TipoDoc>Software provider identifier</TipoDoc>
  </InformacionReferencia>
</Invoice>
```

**5. 5-Year Retention Requirement**
- XML files must be retained for 5 years
- Digital signatures must remain verifiable
- Audit trail must be maintained
- Hacienda can request inspection anytime

**6. Real-Time Submission Requirement**
- Invoice must be submitted within 24-48 hours of issuance
- Hacienda responds with acceptance/rejection
- Status must be tracked and logged
- Rejected invoices must be corrected and resubmitted
